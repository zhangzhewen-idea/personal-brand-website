工程分层与数据流

页面层只负责渲染；数据来源可以从 mock 平滑切换到真实 API。

推荐架构：适度分层 + 可切换数据源
展示层

pages / components / layouts

五类页面、公共导航与页脚、页面区块、基础 UI 组件。只消费 Store 状态，不关心数据来自 mock 还是后端。

↓
状态层

stores

Pinia 按业务域保存基本信息、视频、素材、矩阵账号、课程与用户展示状态。

↓
业务编排层

services

提供页面需要的查询方法，组合业务实体与静态展示配置，统一处理默认值和数据格式。

↓
当前启用

mocks

驼峰字段，严格匹配 database.sql 对应实体；无需网络即可稳定展示。

未来切换

api

独立 axios client、业务请求模块与 DTO 映射；不放入 utils。

共享契约

types/models 定义数据库实体的驼峰类型；configs 存放原型额外的颜色、图标、文案和流程等展示信息。

边界原则

  • 组件不直接导入 mock，也不直接调用 axios。
  • 数据库实体字段与展示配置分离，避免未来 API 模型被原型字段污染。
  • 接入后端时仅替换 Service 使用的数据源，页面与 Store 接口保持稳定。
